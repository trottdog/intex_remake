# Phase 0 Inventory and Schema Understanding

## Scope

Phase 0 establishes one shared understanding of the raw data so later pipelines can reuse the same table map, join logic, date coverage, and leakage rules.

The full 17-table raw dataset is now mirrored into `ml/data/raw/`. The reproducible machine-readable outputs for this phase are generated by `ml/scripts/run_phase0_inventory.py` and written to `ml/reports/tables/`.

## Core Relationship Map

### Donor Domain

- `supporters 1 -> many donations`
- `donations 1 -> many donation_allocations`
- `donations 1 -> many in_kind_donation_items` for in-kind rows only
- `donations many -> 0/1 social_media_posts` via `referral_post_id`
- `donation_allocations many -> 1 safehouses`

### Case Management Domain

- `safehouses 1 -> many residents`
- `residents 1 -> many process_recordings`
- `residents 1 -> many home_visitations`
- `residents 1 -> many education_records`
- `residents 1 -> many health_wellbeing_records`
- `residents 1 -> many intervention_plans`
- `residents 1 -> many incident_reports`

### Outreach and Operations Domain

- `partners 1 -> many partner_assignments`
- `partner_assignments many -> 1 safehouses`
- `safehouses 1 -> many safehouse_monthly_metrics`
- `public_impact_snapshots` is a downstream reporting table, not a source-table join anchor

## Target Label Candidates for the Six Planned Pipelines

| Pipeline | Candidate label | Recommended grain | Primary tables |
| :---- | :---- | :---- | :---- |
| Donor Churn / Lapse Risk | No donation in the next 180 days after a supporter snapshot | Supporter snapshot month | `supporters`, `donations` |
| Reintegration Readiness | `reintegration_status` becomes `Completed` within the next 90 days | Resident monthly snapshot | `residents`, `process_recordings`, `home_visitations`, `education_records`, `health_wellbeing_records`, `intervention_plans` |
| Social Post -> Donation | `donation_referrals > 0` or positive `estimated_donation_value_php` | Social post | `social_media_posts`, `donations` |
| Resident Regression / Elevated Risk | Incident in next 30 to 60 days or worsening risk state | Resident monthly snapshot | `residents`, `incident_reports`, `process_recordings`, `education_records`, `health_wellbeing_records` |
| Campaign Effectiveness Explanation | Donation lift, donor count, average gift, recurring share by campaign period | Campaign-period aggregate | `donations`, `donation_allocations`, `social_media_posts` |
| Intervention Effectiveness Explanation | Post-plan change in incidents, wellbeing, education, or reintegration progress | Resident intervention window | `intervention_plans`, `process_recordings`, `incident_reports`, `education_records`, `health_wellbeing_records` |

## Leakage Risk Notes

- `reintegration_status`, `reintegration_type`, and `date_closed` should be labels or late-stage outcomes, not default features.
- `current_risk_level` can leak if it is recorded after the snapshot used for prediction; it needs time-aware filtering.
- `resolution_date`, `resolved`, and `response_taken` leak incident outcomes when predicting incident occurrence.
- `donation_referrals` and `estimated_donation_value_php` are direct downstream outcomes for social post conversion; use them only as labels.
- `public_impact_snapshots` is a published downstream summary and should stay out of modeling features entirely.
- `session_narrative`, `follow_up_actions`, `observations`, `notes`, and `notes_restricted` can contain future information and sensitive content, so they should remain excluded until a privacy and leakage review is complete.
- Same-period rollups such as `safehouse_monthly_metrics.incident_count` leak if used to predict outcomes for that same month.
- Any feature built from donations, incidents, or progress after the chosen snapshot date must be excluded even if the column itself looks harmless.

## Phase 0 Exit Criteria

- We know the grain and join path for every raw table.
- We know which tables are safe entity anchors versus event streams that need aggregation.
- We know the available date coverage for time-aware splitting.
- We have candidate labels for all six target pipelines.
- We have an explicit first-pass leakage checklist before feature engineering begins.
