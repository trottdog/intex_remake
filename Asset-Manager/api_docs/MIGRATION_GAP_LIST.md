## CRITICAL Risks

> Failures in this category will break the frontend immediately, silently corrupt data, or create security vulnerabilities on day one of migration. These must be fully resolved before any .NET route goes live.

---

### C-01 · CORS Will Block All Cross-Origin Requests

| Property | Detail |
|---|---|
| **Affected area** | Infrastructure / Configuration |
| **Files / Endpoints** | `app.ts` (CORS config); all 154 routes |
| **Why risky** | In the current Replit deployment, frontend and backend share the same host — CORS is not exercised. After migration, the Vercel frontend (e.g., `https://beacon.vercel.app`) and Azure backend (e.g., `https://beacon-api.azurewebsites.net`) are on completely different origins. Every browser API call will be blocked by the browser's CORS policy unless the backend explicitly allows the Vercel origin. |
| **What must be preserved** | `CORS_ALLOWED_ORIGINS` env var must include the exact Vercel production URL. `credentials: false` must be maintained (frontend uses Bearer tokens, not cookies). Allowed headers must include `Content-Type` and `Authorization`. Allowed methods must include `GET`, `POST`, `PATCH`, `PUT`, `DELETE`, `OPTIONS`. |
| **Suggested mitigation** | Set `CORS_ALLOWED_ORIGINS=https://beacon.vercel.app` (and staging preview URLs) in Azure App Service Application Settings before any route goes live. In ASP.NET Core: `app.UseCors(p => p.WithOrigins(...).WithHeaders("Content-Type","Authorization").WithMethods("GET","POST","PATCH","PUT","DELETE","OPTIONS"))`. Do NOT call `.AllowCredentials()`. |
| **Frontend breakage** | **Yes — 100% of API calls will fail** |
| **Security breakage** | No (if configured correctly) |
| **Data integrity breakage** | No |

### C-02 · JWT Secret Must Match Exactly — Existing Sessions Will Break

| Property | Detail |
|---|---|
| **Affected area** | Authentication |
| **Files / Endpoints** | `middleware/auth.ts` (`signToken`, `requireAuth`); all authenticated routes |
| **Why risky** | JWTs are signed with `JWT_SECRET` using HS256. If the .NET backend uses a different secret (or the same secret but a different encoding), all tokens issued by the Express backend become invalid. Users will be logged out and cannot use in-flight sessions. Additionally, the Express dev environment generates an **ephemeral random secret** if `JWT_SECRET` is not set, meaning tokens from a restarted dev server are always invalid. |
| **What must be preserved** | `JWT_SECRET` must be the same string in both Express and .NET. Algorithm must be HS256. Token expiry is **8 hours** (hardcoded in `signToken()`). JWT payload shape: `{ id, username, email, firstName, lastName, role, isActive, mfaEnabled, lastLogin, supporterId, safehouses }`. |
| **Suggested mitigation** | Store `JWT_SECRET` in Azure Key Vault. Reference it as an App Service environment variable. The .NET backend must use `SymmetricSecurityKey(Encoding.UTF8.GetBytes(secret))` with `SecurityAlgorithms.HmacSha256`. Set `ClockSkew = TimeSpan.Zero` to match Express's exact expiry. Do not set issuer or audience claims (Express does not set them). |
| **Frontend breakage** | **Yes — all authenticated users are logged out during cutover** |
| **Security breakage** | Yes if secret is weaker or leaked |
| **Data integrity breakage** | No |

### C-03 · Password Hash Algorithm Must Be Preserved (bcrypt, Cost 12)

| Property | Detail |
|---|---|
| **Affected area** | Authentication |
| **Files / Endpoints** | `routes/auth.ts` (`POST /api/auth/login`), `routes/users.ts` (`POST /api/users`), `routes/auth.ts` (`POST /api/auth/change-password`) |
| **Why risky** | All existing password hashes in the `users` table are bcrypt hashes with cost factor 12 (generated by `bcrypt.hash(password, 12)`). If the .NET backend uses a different algorithm (ASP.NET Core Identity defaults to PBKDF2) or a different bcrypt library with incompatible output, all existing users will fail login. |
| **What must be preserved** | `BCrypt.Verify(password, storedHash)` in .NET must use bcrypt (not PBKDF2, not Argon2). Install `Bcrypt.Net-Next` NuGet package. Do NOT use `PasswordHasher<T>` from ASP.NET Core Identity (it uses PBKDF2). |
| **Suggested mitigation** | Use `BCrypt.Net.BCrypt.Verify(plaintext, hash)` for login. Use `BCrypt.Net.BCrypt.HashPassword(plaintext, 12)` for new passwords and password changes. |
| **Frontend breakage** | **Yes — existing users cannot log in** |
| **Security breakage** | Yes if weaker algorithm used |
| **Data integrity breakage** | No (hashes stay in DB) |

| **Security breakage** | No |
| **Data integrity breakage** | No |

### C-05 · NUMERIC Columns Must Serialize as JSON Numbers, Not Strings

| Property | Detail |
|---|---|
| **Affected area** | API Contract / Data Types |
| **Files / Endpoints** | `donations.ts` (amount, estimatedValue, totalAllocated, unallocated), `campaigns.ts` (goalAmount, totalRaised), `inKindDonationItems.ts` (quantity, estimatedUnitValue), `donationAllocations.ts` (amountAllocated), `supporters.ts` (lifetimeGiving), `safehouses.ts` (avgHealthScore, avgEducationProgress), and many more |
| **Why risky** | PostgreSQL `numeric`/`decimal` columns arrive from `pg` (Node.js driver) as **strings**. The Express code explicitly calls `parseFloat()` before serializing, converting them to JSON numbers. Npgsql (.NET) maps `numeric` to `decimal` natively — but `System.Text.Json` serializes `decimal` as a JSON number with full decimal precision, which is correct. However, if any code accidentally serializes the raw Npgsql `NpgsqlDataReader` string value, the frontend receives a string where it expects a number, causing `NaN` in calculations. |
| **What must be preserved** | All NUMERIC-derived fields in JSON responses must be JSON numbers (float), not strings. Exception: ML monetary values in `superadminMl.ts` must be strings with 2 decimal places (`.toFixed(2)` behavior). |
| **Suggested mitigation** | Use `decimal` properties in C# DTOs for all monetary fields. Configure `System.Text.Json` to serialize `decimal` normally (default behavior). For ML monetary fields, use `string` properties and format with `value.ToString("0.00")`. |
| **Frontend breakage** | **Yes — financial calculations produce NaN** |
| **Security breakage** | No |
| **Data integrity breakage** | No |

### C-06 · JSONB Columns Must Deserialize to Objects, Not Strings

| Property | Detail |
|---|---|
| **Affected area** | API Contract / Data Types |
| **Files / Endpoints** | `case_sessions.contextJson`, `safehouse_metrics.metricPayloadJson`, `ml_pipeline_runs.featureImportanceJson`, `ml_prediction_snapshots.churnTopDrivers`, `ml_prediction_snapshots.regressionRiskDrivers` — accessed in `superadminMl.ts`, `caseManagement.ts`, `dashboard.ts` |
| **Why risky** | Drizzle ORM automatically parses JSONB columns from PostgreSQL into JavaScript objects. Npgsql returns JSONB as a raw `string`. If the .NET backend passes this string directly to the JSON serializer, the API response contains a **JSON-encoded string** (with escaped quotes) instead of a nested object, breaking any frontend code that accesses nested properties. |

| **Frontend breakage** | **Yes — ML dashboard widgets and case session data break** |
| **Security breakage** | No |
| **Data integrity breakage** | No |

### C-08 · `program_updates` DELETE Has Manual Cascade — No DB-Level Cascade Exists

| Property | Detail |
|---|---|
| **Affected area** | Data Integrity |
| **Files / Endpoints** | `routes/programUpdates.ts` `DELETE /api/program-updates/:id` |
| **Why risky** | The Express code deletes from `donor_viewed_items` before deleting from `program_updates`. There is **no `ON DELETE CASCADE`** in the database. If the .NET backend issues only `DELETE FROM program_updates WHERE update_id = :id`, the `donor_viewed_items` rows become orphaned, violating referential integrity and potentially causing FK constraint errors on future queries. |
| **What must be preserved** | Two-step delete: (1) `DELETE FROM donor_viewed_items WHERE item_type = 'update' AND item_id = :id`, then (2) `DELETE FROM program_updates WHERE update_id = :id`. Both must be in a transaction. |
| **Suggested mitigation** | Either replicate the two-step delete in .NET inside a `BEGIN … COMMIT` transaction, or add `ON DELETE CASCADE` to the `donor_viewed_items.item_id` FK in Supabase before migration. The latter is cleaner and removes code-level dependency. |
| **Frontend breakage** | No |
| **Security breakage** | No |
| **Data integrity breakage** | **Yes — orphaned rows or FK constraint errors** |

### C-09 · Supabase Database Is Accessible Only From Azure — Never From Vercel

| Property | Detail |
|---|---|
| **Affected area** | Infrastructure / Architecture |
| **Files / Endpoints** | All DB queries; `lib/db.ts`; `DATABASE_URL` secret |
| **Why risky** | The Supabase PostgreSQL database must be accessible ONLY from the Azure .NET backend. The Vercel frontend (a static React SPA) must NEVER have direct DB access — all data flows through the API. This is already the case architecturally, but must be explicitly enforced: (1) `DATABASE_URL` must only be set on Azure (App Service environment / Key Vault), never on Vercel. (2) Supabase network restrictions should allow connections only from Azure outbound IPs. |
| **What must be preserved** | Architecture: Vercel → Azure API → Supabase. No Vercel → Supabase direct calls. |
| **Suggested mitigation** | Do not set `DATABASE_URL` in Vercel environment variables. In Supabase, configure "Allowed IP Addresses" to Azure's outbound IP range (or use Supabase's IP allowlist feature). Use Supabase pooler URL (port 6543, `?pgbouncer=true`) for runtime; direct URL (port 5432) for schema migrations only. |
| **Frontend breakage** | No |
| **Security breakage** | **Yes if misconfigured — DB exposed to internet** |
| **Data integrity breakage** | No |

### H-09 · Live `isActive` DB Check on Every Authenticated Request

| Property | Detail |
|---|---|
| **Affected area** | Authentication / Performance |
| **Files / Endpoints** | `middleware/auth.ts` `requireAuth` — all 140+ authenticated routes |
| **Why risky** | Express's `requireAuth` queries the `users` table on EVERY authenticated API call to confirm `isActive = true`. This is a round-trip per request. .NET's default `JwtBearerHandler` does not do this — it trusts the JWT payload. If .NET omits this check, a deactivated user account can continue making API calls until the token expires (up to 8 hours). |
| **What must be preserved** | Deactivated accounts (`isActive = false`) must be rejected on every request, not only at next login. |
| **Suggested mitigation** | Implement a custom `IClaimsTransformation` or middleware that checks `isActive` from a short-TTL cache (Redis). Cache TTL of 60–300 seconds provides a reasonable revocation window without the N+1 DB round-trip. |
| **Frontend breakage** | No |
| **Security breakage** | **Yes — disabled accounts remain functional for up to 8 hours** |
| **Data integrity breakage** | No |

### H-13 · All Error Responses Must Be JSON — No HTML Fallbacks

| Property | Detail |
|---|---|
| **Affected area** | API Contract |
| **Files / Endpoints** | All endpoints; global error handling |
| **Why risky** | ASP.NET Core's default exception handler returns an HTML error page for unhandled exceptions. The frontend's `fetch` wrapper expects `response.json()` on all responses. If a 500 returns HTML, `JSON.parse` throws in the frontend, usually producing a cryptic "unexpected token" error. | **Frontend breakage** | **Yes — any unhandled exception breaks frontend error handling** |
| **Security breakage** | No |
| **Data integrity breakage** | No |

### H-14 · No Transactions in Express — .NET Must Add Them

| Property | Detail |
|---|---|
| **Affected area** | Data Integrity |
| **Files / Endpoints** | `routes/auth.ts` (register: user + assignments), `routes/users.ts` (create/patch: user + assignments), `routes/programUpdates.ts` (delete: viewed items + update) |
| **Why risky** | The Express backend performs multi-step writes (e.g., insert user then insert safehouse assignments) without transactions. A failure between the two steps leaves the DB in a partial state. While this has not caused known issues today, replicating the same pattern in .NET would perpetuate a data integrity risk. The migration is the correct moment to fix this. |
| **What must be preserved** | The net effect of each multi-step operation must be atomic. |
| **Suggested mitigation** | Wrap all multi-step writes in .NET in `using var tx = await conn.BeginTransactionAsync(); ... await tx.CommitAsync();`. Specifically: user create/update + assignment sync, program update delete. |
| **Frontend breakage** | No |
| **Security breakage** | No |
| **Data integrity breakage** | **Yes — partial states possible on failure** |

## Executive Summary for Rebuilding the Backend Outside Replit

This section is written for a development team taking over the .NET implementation without Replit access.

### What You Are Building

A .NET Web API backend hosted on **Azure App Service** that serves a **React + TypeScript SPA hosted on Vercel**. The database is **PostgreSQL on Supabase**. The frontend communicates entirely via authenticated REST API calls using JWT Bearer tokens. There is no server-side rendering, no WebSocket layer, and no file upload infrastructure.

### The Three Non-Negotiable Contracts

**1. JWT must be HS256, signed with the same secret as Express, 8-hour expiry.**
All existing users have active tokens. If the secret or algorithm changes, everyone is logged out on cutover. Store the secret in Azure Key Vault and inject as an app setting. Use `Bcrypt.Net-Next` for password hashing (not ASP.NET Core Identity's PBKDF2).

**2. The JSON response envelope for standard list endpoints must be exactly `{ "data": [...], "total": N, "pagination": { "page": N, "pageSize": N, "totalPages": N, "hasNext": bool, "hasPrev": bool } }`.**
Every frontend list view depends on this shape. Any deviation breaks list views simultaneously.

**3. CORS must be configured to allow the Vercel frontend domain before the first route goes live.**
Without this, 100% of browser API calls fail with a CORS error. Set `CORS_ALLOWED_ORIGINS` to the Vercel production URL in Azure App Service Application Settings.

### The Five Most Dangerous Data Type Traps

1. **NUMERIC → float**: PostgreSQL `numeric` columns → Npgsql `decimal` → must serialize as JSON numbers (not strings). Except: ML monetary fields → must serialize as strings with 2 decimal places.
2. **JSONB → object**: Npgsql returns JSONB as `string`. Deserialize with `System.Text.Json` before returning.
3. **staffSafehouseAssignments.userId is VARCHAR**: Always cast the integer userId to string in queries.
4. **safehouses.name (not safehouseName)**: The DB column is `name`. Map explicitly in EF Core or use raw SQL.
5. **totalRaised, totalAllocated, unallocated**: These are computed at query time, not stored columns. Implement as SQL subqueries.

### The Five Most Dangerous Authorization Traps

1. **GET /api/impact-snapshots is public** (no auth). GET /api/admin/impact-snapshots requires auth. Both exist.
2. **GET /api/social-media requires auth** despite being in `public.service.ts` on the frontend.
3. **ML watchlists must exclude** residents with `ml_scores_restricted = true`.
4. **ML endpoints must redact donor emails** to `"***@***.***"` for the `admin` role.
5. **program_updates DELETE must first delete from `donor_viewed_items`** — no DB cascade exists.

### What the Frontend Assumes About Error Responses

Every error response — 400, 401, 403, 404, 429, 500 — must be `application/json` with the body `{ "error": "some message" }`. If any error returns HTML (ASP.NET Core default exception page), the frontend `response.json()` call throws, producing a cryptic "unexpected token" error in the UI. Configure `app.UseExceptionHandler()` with a custom JSON formatter as one of the very first infrastructure tasks.

### Deployment Sequence for Zero Downtime Cutover

1. Deploy .NET to Azure without changing `VITE_API_BASE_URL` in Vercel (Express still active).
2. Validate all 154 routes against .NET using the Express codebase as the ground truth.
3. Update `VITE_API_BASE_URL` in Vercel to point to Azure (one env var change triggers a Vercel redeploy).
4. Monitor for 48 hours; keep Express running as a fallback.
5. Decommission Express after confidence period.
